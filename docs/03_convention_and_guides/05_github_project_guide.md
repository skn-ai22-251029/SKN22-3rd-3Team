# 🐙 GitHub Project 관리 가이드

이 문서는 `roadmap_backlog.md`를 실제 **GitHub Projects/Issues**로 옮겨서 관리하는 방법과 용어를 설명합니다.

---

## 1. 핵심 용어 설명 (Concepts)

### ⛳️ 마일스톤 (Milestone)
*   **의미**: 프로젝트의 **"중간 목표 지점"** 또는 **"큰 챕터"**입니다.
*   **우리 프로젝트 매핑**: 로드맵의 **`Phase`** 가 바로 마일스톤입니다.
*   **역할**: 여러 개의 이슈(할 일)들을 하나로 묶어 관리하며, **마감 기한(Due Date)** 과 **진척도(%)** 를 보여줍니다.
    *   *예시: "Phase 1: V3 안정화" (진행률 50% - 이슈 10개 중 5개 완료)*

### 🎫 이슈 (Issue)
*   **의미**: 실제로 해결해야 할 **"작업 단위"** 하나하나입니다.
*   **우리 프로젝트 매핑**: 로드맵의 **체크박스 항목(`- [ ]`)** 하나가 이슈 1개입니다.
*   **역할**: 누가 할지(`Assignee`), 언제까지 할지, 어떤 마일스톤에 속하는지 정의합니다.
    *   *예시: "지역 코드 정합성 검증"*

### 🏷️ 레이블 (Label)
*   **의미**: 이슈의 성격을 나타내는 **"꼬리표"**입니다.
*   **추천 레이블**:
    *   `🐞 Bug`: 버그 수정
    *   `✨ Feature`: 새로운 기능 추가
    *   `📚 Docs`: 문서 작업
    *   `💬 Discussion`: 논의가 필요한 주제 (V4 스키마 등)
    *   `🔥 Critical`: 긴급/중요 항목

---

## 2. GitHub 등록 및 작업 흐름 (Workflow)

### Step 1. 마일스톤 만들기 (Milestones)
GitHub 리포지토리의 `Issues` > `Milestones` 탭에서 생성합니다.

1.  **Phase 1: V3 안정화 및 버그 박멸** (기한 설정: 예 - 2월 28일)
2.  **Phase 2: UI/UX 고도화 & 개인화** (기한 설정: 예 - 3월 15일)
3.  **Phase 3: 확장 및 자동화** (기한 설정: 미정)

### Step 2. 이슈 만들기 (Issues)
`roadmap_backlog.md`의 내용을 복사해서 이슈를 만듭니다.

1.  제목: `[Phase 1] 통합 테스트 시나리오 설계 및 검증`
2.  내용: `test_plan.md 작성`, `Liaison 테스트` 등 세부 할 일 기재.
3.  **우측 사이드바 설정 (중요!)**:
    *   **Assignees**: 담당자 지정 (본인 등)
    *   **Milestone**: `Phase 1: V3 안정화...` 선택
    *   **Labels**: `Task`, `Test` 등 선택

### Step 3. 작업 및 완료
1.  작업을 완료하고 PR을 올릴 때 `Resolves #이슈번호`를 적습니다.
2.  PR이 머지되면 이슈가 자동으로 닫히고(Close), 마일스톤의 진행률 게이지가 올라갑니다! 📈

## 3. 요약 (Mapping Table)

| Roadmap (문서) | GitHub (시스템) |
| :--- | :--- |
| **Phase 1, 2, 3** | **Milestone (마일스톤)** |
| **체크박스 항목** (`- [ ]`) | **Issue (이슈)** |
| **Critical, Discussion** | **Label (레이블)** |
| **담당자** | **Assignee** |

이 규칙대로만 등록하시면, GitHub에서 **"우리가 어디까지 왔는지"** 한눈에 보실 수 있습니다!
